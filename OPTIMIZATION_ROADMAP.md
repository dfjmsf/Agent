# ASTrea 长线优化路线图 — 全组件 A+ 之路

> 创建时间：2026-04-10
> 目标：将 ASTrea 的每个核心组件从当前水平提升至 A / A+ 级别
> 更新策略：每完成一个里程碑后更新对应组件的状态

---

## 总览：当前评级与目标

| 组件 | 当前评级 | 核心短板 | 目标评级 | 关键突破口 |
|------|---------|---------|---------|-----------|
| **Coder** | B+ | 偶尔忽略铁律、大 prompt 溢出 | A | 上下文精准裁剪 + 增量编辑常态化 |
| **Reviewer** | B+ | 缺少跨文件语义校验 | A | AST 显微镜全面接管 L0 |
| **Manager** | B | 合同自相矛盾、大项目拆分不精准 | A | 合同一致性自检 + 模块组智能排序 |
| **PM** | C+ | 项目认知浅、无主动检索能力 | A | Skill 赋能（Web Search + 文件探索） |
| **QA** | C- | 盲狙、无动态诊断 | A+ | ReAct Tool Calling + 实时终端交互 |
| **Engine** | B+ | 体量过大、降级策略粗糙 | A | 职责拆解 + 智能降级分级 |
| **Memory** | B | 召回精度不够、冷启动空白 | A | 向量检索 + 全局经验蒸馏 |
| **Sandbox** | B | 前端验证薄弱、资源清理偶有残留 | A | Playwright 集成 + 进程树健壮管理 |

---

## 各组件升级路线

---

### 1. Coder：B+ → A

**现状**：单文件生成能力扎实，Playbook + Fix Mode 覆盖主要场景。

**短板**：
- 大项目 prompt 超窗口时会丢失铁律
- 全量重写模式浪费 token，增量编辑（Fix Mode）仅在重试时触发
- 前端文件（HTML/CSS/JS）的 UI 美观度不稳定

**升级路径**：

| 阶段 | 措施 | 效果 |
|------|------|------|
| **A-1** | vfs_context 智能裁剪：只注入与当前文件有直接依赖关系的代码，而非所有"相关文件" | prompt 瘦身 40%，铁律不再被挤掉 |
| **A-2** | 增量编辑常态化：首轮也走 Skeleton → Fill 两阶段，减少一次性全文输出的失控风险 | 结构稳定性提升 |
| **A-3** | 前端 UI Playbook 强化：为 CSS 注入设计系统规范（配色方案、间距、圆角、阴影等） | 生成的 UI 从"能跑"到"好看" |
| **A-4** | Tree-sitter 微观切片：Patch Mode 下只向 LLM 发送需要修改的函数/类，而非整个文件 | Patch 精度从文件级到函数级 |

---

### 2. Reviewer：B+ → A

**现状**：L0 确定性传感器矩阵 + L1 LLM 轻量审查，已覆盖路由/变量/模板一致性。

**短板**：
- L0 传感器只检查单文件内的约定，不做跨文件语义校验
- 缺少对前端 JS/TS 的 AST 分析能力
- 无法检测"功能正确但性能灾难"的代码

**升级路径**：

| 阶段 | 措施 | 效果 |
|------|------|------|
| **A-1** | 跨文件合同校验：Reviewer 拿到 Manager 规划书 + 所有已完成文件的接口快照，做全局一致性检查 | 拦截 Manager 合同歧义导致的「A 文件调 B 文件不存在的函数」 |
| **A-2** | 前端 AST 分析：用 Tree-sitter 解析 JS/TS/Vue/JSX，检查 fetch URL 是否与后端路由一致 | 前后端接口不匹配的检出率从 60% → 95% |
| **A-3** | 安全审计传感器：检测 SQL 注入（字符串拼接 SQL）、XSS（v-html / dangerouslySetInnerHTML）等 | 零成本安全兜底 |

---

### 3. Manager：B → A

**现状**：能拆分任务、生成规划书、做 mini re-plan。

**短板**：
- `api_contracts` 和 `module_interfaces` 偶尔自相矛盾
- 大项目的模块组拆分不够智能（依赖顺序有时错误）
- 对用户原始需求的理解依赖 PlannerLite 的低质量转述

**升级路径**：

| 阶段 | 措施 | 效果 |
|------|------|------|
| **A-1** | 合同自检层：Manager 输出规划书后，运行一个确定性校验器检查 api_contracts.path 与 page_routes.path 是否一致 | 消灭 90% 的合同歧义 | **✅ 已完成** |
| **A-2** | 依赖图排序：用 DAG 拓扑排序任务执行顺序，而非让 Manager 凭直觉排列 | 消灭"先写 routes.py 但 models.py 还没写"的问题 |
| **A-3** | 接入 plan.md 原文：Manager 直接读取用户审批过的 plan.md，而非 PlannerLite 的二手转述 | 信息保真度提升 |
| **A-4** | 增量规划：Patch Mode 下 Manager 只规划变更部分，而非重新扫描全部文件 | Patch 速度提升 3x |

---

### 4. PM：C+ → A

**现状**：能理解需求、路由意图、展示方案。但对项目缺乏深度认知。

**短板**：
- 无主动信息获取能力（不能搜索文档、不能浏览代码）
- 对 Patch Mode 的项目现状几乎无感知
- 对冷门技术栈（如 Prisma、tRPC）完全靠猜

**升级路径**：

| 阶段 | 措施 | 效果 |
|------|------|------|
| **A-1** | 项目感知注入：Patch Mode 下将项目文件树 + 关键文件摘要注入 PM 上下文 | PM 不再对已有项目"失明" |
| **A-2** | Web Search Skill：赋予 PM 搜索能力，遇到不认识的库/框架时自行查文档 | 冷门技术栈的成功率从 30% → 70% |
| **A-3** | 多轮需求澄清：PM 主动提问模糊需求（"你说的列表页需要分页吗？"），而非默认补全 | 用户意图对齐度提升 |
| **A-4** | 项目历史记忆：PM 能回忆同一个项目过去的修改历史和用户偏好 | 长期项目的体验飞跃 |

---

### 5. QA：C- → A+

**现状**：纯脚本盲狙，无动态交互能力。

**短板**：
- 不能看 stderr、不能装依赖、不能交互式调试
- 反馈给 Coder 的信息太粗糙（"服务无法启动" ≈ 废话）
- 前端只能做 npm build 检查，不能验证页面渲染

**升级路径**：

| 阶段 | 措施 | 效果 |
|------|------|------|
| **A-1** | ReAct QA Agent：持 run_terminal / http_request 等 Skill 下场实时测试 | 从盲狙到有眼睛的测试 | **✅ 已完成** |
| **A-2** | 精准归因反馈：QA 的 report 包含具体文件名 + 行号 + 修复建议 | Coder 修复效率从 40% → 80% |
| **A-3** | Playwright 视觉验证：用无头浏览器打开页面，截图检查 UI 是否正常渲染 | 前端验证从"能编译"到"能看" |
| **A-4** | 回归测试记忆：QA 记住上一轮测了什么、哪些 API 通过了，Patch 后只测变更相关的 | 增量测试，速度提升 5x |

---

### 6. Engine：B+ → A

**现状**：能协调全流程，有熔断和降级机制。

**短板**：
- 2100 行单文件，职责过多
- 降级策略粗糙（只有"通过"和"警告交付"两档）
- Create 和 Patch 模式的代码重复度较高

**升级路径**：

| 阶段 | 措施 | 效果 |
|------|------|------|
| **A-1** | 职责拆解：将 Rollback、Git 操作、集成测试调度抽取为独立模块 | 可维护性大幅提升 |
| **A-2** | 智能降级分级：success / warning / partial_fail / fail 四档，前端差异化展示 | 用户对项目状态的感知更精准 |
| **A-3** | DAG 并行执行：无依赖关系的 task 可以并行给 Coder | 10 文件项目速度提升 2-3x |
| **A-4** | 断点续传：Engine 崩溃后从 Checkpoint 恢复，而非从零开始 | 鲁棒性质的飞跃 |

---

### 7. Memory：B → A

**现状**：有 Synthesizer + Auditor，能提炼和归因经验。

**短板**：
- 基于 FTS5 的关键词召回精度有限
- 冷启动时完全无经验可用
- 全局经验和项目经验没有清晰的分层召回策略

**升级路径**：

| 阶段 | 措施 | 效果 |
|------|------|------|
| **A-1** | 向量检索增强：embedding + cosine 相似度，替代纯 FTS5 关键词匹配 | 语义相关的经验也能被召回 |
| **A-2** | 经验蒸馏：定期将低频重复经验合并为更通用的规则 | 消除经验库膨胀 |
| **A-3** | 预置全局知识库：将常见框架的踩坑经验预埋（冷启动不再裸奔） | 首次使用即有 50+ 条经验 |

---

### 8. Sandbox：B → A

**现状**：venv 隔离 + 自动依赖安装 + npm 构建支持。

**短板**：
- Windows 进程树清理偶有残留
- 前端验证只有 npm build，不能检查渲染结果
- sandbox 的 token 超时控制有时不精准

**升级路径**：

| 阶段 | 措施 | 效果 |
|------|------|------|
| **A-1** | 进程守护增强：用 Job Object (Windows) 确保子进程完全回收 | 零残留进程 |
| **A-2** | Playwright 沙盒集成：sandbox 内启动无头浏览器验证前端 | QA 的 A-3 依赖此项 |
| **A-3** | 资源配额：CPU/内存/磁盘使用上限，防止恶意代码耗尽系统资源 | 安全性质的飞跃 |

---

## 推荐攻坚顺序

```
Phase 1 (当务之急):  QA A-1 → Manager A-1 → Coder A-1
                     ↑ 拔掉最大瓶颈    ↑ 合同不再矛盾   ↑ prompt 不再溢出

Phase 2 (巩固提升):  PM A-1 → Engine A-1 → Reviewer A-1
                     ↑ Patch Mode 不再盲  ↑ 代码可维护    ↑ 跨文件拦截

Phase 3 (精益求精):  QA A-3 → Memory A-1 → Coder A-4
                     ↑ 视觉验证         ↑ 语义召回       ↑ 函数级编辑

Phase 4 (极致性能):  Engine A-3 → QA A-4 → Sandbox A-2
                     ↑ 并行执行         ↑ 增量测试        ↑ 浏览器沙盒
```
