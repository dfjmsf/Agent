## 前端 UI 设计引导（P0 - 底线约束）

这份补丁的职责不是给出一套写死的视觉参数表，而是为 Coder 提供稳定的设计方向、退化约束和常见反模式提醒。

请把它理解为：

- 需要遵守的是设计原则和质量下限
- 不需要照抄固定色值、固定字号、固定圆角、固定组件库
- 需要根据项目场景主动做出有理由的视觉选择

### 1. 先确定视觉方向，再开始写页面

生成前端时，不要一上来就堆 DOM 和表单。先判断这个项目更像哪一类产品：

- 工具型后台
- 数据型仪表盘
- 内容展示页
- 表单工作台
- 营销着陆页
- 轻量作品页

不同类型的页面，配色、布局、留白和信息密度应明显不同。不要所有项目都落成同一种蓝白默认后台。

### 2. 优先组织设计 token，而不是到处散落样式

优先通过 CSS 变量或等价的主题 token 组织这些视觉要素：

- 背景层次
- 表面层级
- 主文本与辅助文本
- 边框与分隔
- 主强调色与状态色
- 阴影与圆角
- 间距节奏

重点不是变量名必须统一，而是页面的视觉规则要“集中定义、统一引用”，避免颜色、阴影、边框在各处零散硬编码。

### 3. 字体要有气质，不要裸用浏览器默认感

字体不必固定为某一款，但应该有明确判断：

- 这是偏产品感、编辑感、数据感，还是偏内容感
- 标题和正文是否存在清晰层级
- 数字、金额、统计值是否需要更稳定的视觉呈现

避免出现：

- 整页只有浏览器默认字体
- 标题、正文、辅助文案几乎没有层级差异
- 大量数据区仍然像普通说明文排版

### 4. 页面必须有主次结构，不能像平铺文档

无论页面大小，都应该先建立信息主次：

- 哪个区域是首屏焦点
- 哪个区域是主操作区
- 哪个区域是补充信息区
- 哪个区域是列表、统计或详情区

避免出现：

- 整页只有一个大容器把所有内容从上到下堆完
- 卡片、表格、表单都在同一视觉权重上
- 页面没有节奏，像文档排版而不像产品界面

### 5. 可交互元素必须给出反馈

按钮、链接、可点击卡片、标签筛选、切换器等元素，至少应体现出一种明确反馈：

- hover
- focus
- active
- selected
- disabled

重点不是必须使用哪种动画，而是让用户看得出哪里可以操作、操作后发生了什么。

### 6. 状态设计不能缺席

页面不是只在“数据正常加载完成”这一刻存在。优先考虑至少一种状态表现：

- 空状态
- 加载状态
- 禁用状态
- 错误提示
- 成功反馈

不要把状态处理留成一句裸文本，优先让状态区也保持与页面主视觉一致。

### 7. 反模式约束

以下输出应主动避免：

- 纯白背景加纯黑文字的“文档页”气质
- 所有区域只靠细边框硬切，没有表面层次
- 表格、表单、卡片混用不同语言，像三个页面拼起来
- 只顾堆功能，没有首屏视觉重点
- 随机使用高饱和颜色，导致页面像模板市场拼装品
- 每种按钮都长得差不多，看不出主次
- 通过大量装饰去掩盖信息结构混乱

## 前端 UI 设计引导（P1 - 高频落地规则）

### 8. 为项目选一条明确的视觉路线

在动手实现前，先从场景出发选择一条视觉路线，再围绕这条路线组织页面。

可参考的方向：

- 工具 / 效率类：克制、中性、结构清晰、强调秩序
- 数据 / 仪表盘类：层级明显、统计卡突出、信息密度适中
- 教育 / 知识类：温和、可信、内容可读性优先
- 健康 / 生活类：更轻、更柔和，有适度亲和力
- 内容 / 品牌展示类：首屏更有视觉情绪，分区更鲜明

不要一律套成“中间一个白卡片 + 上面一个大标题 + 下面一堆按钮”的默认模板。

### 9. 桌面端布局先解决信息组织

当前目标以桌面端体验为主，因此布局重点是信息组织，而不是移动端折叠。

优先明确：

- 主内容区是否需要居中容器还是全宽工作区
- 是否存在左右分栏，哪边是主视图、哪边是辅助区
- 列表页是否需要筛选区、工具栏、统计摘要
- 仪表盘是否需要先给关键指标，再给趋势或明细
- 表单页是否要拆为若干逻辑分组，而不是一串字段直排

推荐思路：

- 仪表盘：摘要卡片在前，核心图表或主列表在中，辅助信息在后
- 列表页：标题区、操作区、筛选区、结果区分层清晰
- 详情页：概要信息与详细信息分组，避免整页长文
- 表单页：分组、说明、操作按钮应有明确关系

### 10. 配色不求复杂，但必须成体系

配色不需要炫技，但要形成清晰角色分工：

- 背景色负责情绪基调
- 表面色负责承载内容
- 文字色负责信息可读性
- 强调色负责交互与重点
- 状态色负责成功、警告、错误、提示

更重要的是关系，而不是具体数值。只要能看出“这是一套体系”，就比到处随手挑色强很多。

避免以下情况：

- 整页只有一种底色，所有模块挤在一起
- 强调色同时承担按钮、标签、链接、边框、标题，导致主次混乱
- 颜色种类太多，每块区域都像单独设计

### 11. 组件外观需要统一语言

同一页面内，组件应该像一家产品里的成员，而不是临时拼装。

应重点统一：

- 按钮的层级关系
- 输入框、选择器、搜索框的边框与表面语言
- 卡片的边界、阴影、标题区和内容区组织方式
- 标签、徽标、提示条的视觉强度
- 表格、列表、分组面板的密度与留白

建议：

- 主按钮更明确，次按钮更克制，辅助按钮更轻
- 卡片可有轻表面感，但不要厚重到喧宾夺主
- 表格需要行层次、表头层次和操作区节奏
- 筛选栏不只是把控件摆上去，还要像一个完整操作区

### 12. 标题体系要拉开层级

页面通常至少应区分：

- 页面级标题
- 分区级标题
- 卡片级标题
- 正文文本
- 辅助说明
- 弱提示信息

避免：

- 所有标题都差不多大
- 辅助文案和正文一样重
- 大段说明文字没有分组

设计目标不是“字越大越高级”，而是让用户能快速扫出信息结构。

### 13. 留白和分隔要服务结构，而不是装饰

留白的作用是让结构被看见，而不是让页面变空。

优先处理：

- 模块之间的呼吸感
- 标题和正文之间的距离
- 操作区与结果区之间的切换感
- 卡片内部的内容分层

不要：

- 什么都贴得很紧，像管理系统截图
- 什么都隔得很远，导致页面支离破碎
- 到处堆阴影、描边、渐变来假装有层次

### 14. 动效只做增强，不做主角

动效存在的目的，是让状态变化更自然、交互更顺滑。

适合加入的动效：

- hover 过渡
- 面板显隐
- 标签切换
- 数据区域淡入
- 按钮状态变化

不建议：

- 大面积持续运动背景
- 每个组件都在浮动、弹跳或旋转
- 把缺少层次的问题交给动效来掩盖

### 15. 状态区也要纳入设计系统

这些区域不要随意处理：

- 空状态
- 无搜索结果
- 表单校验提示
- 成功保存反馈
- 错误提醒
- 数据加载中

好的做法是：

- 状态区仍延续页面主色与层级
- 状态文案简洁、明确、可行动
- 如有必要，可配合轻图形、图标或说明文本

不要只留下一句突兀的“暂无数据”或一行红字报错。

## 前端 UI 设计引导（P2 - 参考模板与启发）

下面的内容是参考起点，不是固定模板。可以借用结构、语气和组织方式，但不要机械照抄。

### 16. 语义化 token 骨架参考

```css
:root {
  --bg: #f4f1ea;
  --surface: #fffdf8;
  --surface-strong: #f0eadf;
  --text: #1f1d1a;
  --text-muted: #6e675c;
  --line: #d8d0c4;
  --accent: #2f6fed;
  --accent-contrast: #ffffff;
  --success: #1f8a5b;
  --warning: #b7791f;
  --danger: #c75050;
  --shadow-soft: 0 10px 30px rgba(31, 29, 26, 0.08);
  --radius-card: 18px;
  --radius-control: 12px;
}
```

要点不是使用这一套颜色，而是：

- token 语义化
- 颜色角色清楚
- 表面层次、文本层次、强调色分工明确

### 17. 仪表盘页面参考方向

适合采用：

- 顶部标题区：标题、描述、主操作
- 第一屏统计卡：3 到 5 个关键信息块
- 第二屏主内容：图表、趋势、重点列表二选一或并列
- 侧边或下方补充：提醒、待办、最近活动

视觉感觉应偏：

- 强调层级
- 控制信息密度
- 卡片之间有节奏

不应：

- 第一屏直接塞满表格
- 所有卡片一样大、一样重
- 强调色把每一块都涂满

### 18. 列表 + 筛选页参考方向

推荐结构：

- 页面标题区：标题、说明、主要动作
- 工具条：搜索、筛选、排序、批量操作
- 结果摘要：总量、当前条件、快速标签
- 数据区：表格或卡片列表

体验目标：

- 用户一眼能看懂“能做什么”和“现在看到什么”
- 操作区和结果区有明显切分
- 表头、行 hover、操作按钮都属于同一语言

### 19. 表单工作台参考方向

推荐结构：

- 顶部标题与说明
- 分组表单区
- 关键字段优先展示
- 次要字段折叠或放后
- 底部或右上保留明确提交动作

体验目标：

- 表单不是字段堆砌，而是有流程感
- 帮助文字存在，但不过量
- 主按钮清晰可见，不需要到处找

### 20. 展示页 / Landing 参考方向

可以采用：

- Hero 区：一句核心价值 + 一句说明 + 主要 CTA
- Feature 区：三到六个价值点
- Proof 区：数据、评价、案例或产品截图
- Final CTA：收尾行动区

视觉上应更重：

- 情绪与节奏
- 分区对比
- 标题张力

但不要：

- 使用廉价的“渐变铺满 + 发光按钮 + 一堆悬浮圆球”
- 为了显得高级而牺牲信息清晰度

### 21. 快速自检清单

生成完成后，先快速问自己：

- 这页一眼能看出主功能吗？
- 配色像一个产品，而不是几个组件拼盘吗？
- 页面标题、分区标题、正文、辅助文案有层级吗？
- 主按钮和次按钮的权重分清了吗？
- 表格、表单、卡片是不是同一种设计语言？
- 页面更像产品界面，而不是文档、表格导出页或脚手架初始页吗？

如果这些问题大多答不上来，就继续优化，不要急着交付。
